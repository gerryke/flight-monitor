# 设计：接入去哪儿数据源 + 最低价突出显示

日期：2026-06-07

## 背景与目标

当前 `fetch_prices.py` 只从携程抓取目标航班的最低含税价。本次要新增「去哪儿」作为
第二个抓取来源，并在网站上把携程价、去哪儿价并列展示，**重点突出两者中更低的那个价格**。

## 范围

- 抓取改动**仅限本机脚本** `scripts/fetch_prices.py`（不改云端 routine `scripts/update.md`）
- 数据合并/快照逻辑（`scripts/pricedata.py`）调整为支持「同一航班+日期下多个来源价格并存」
- 前端「当前最低价」表格与「历史价格趋势」折线图相应调整
- 开工前先解决一个遗留问题：测试套件当前有 1 个失败用例（见第 4 节）

## 1. 数据模型变更

记录结构本身不变，仍是 `{flight, depart_date, ts, price_cny, source}`，`source`
新增取值 `"去哪儿"`（与现有 `"携程"` 风格一致）。需要调整的是两处分组粒度：

- **`pricedata.merge` 的去重键**：从 `(航班, 日期, 整点小时)` 改为
  `(航班, 日期, 整点小时, 来源)`。否则同一小时内携程和去哪儿各自抓到的记录会被
  当成"同一条"，互相覆盖，丢掉一方的数据点。
- **`pricedata.build_latest` 的分组键**：从 `(航班, 日期)` 改为
  `(航班, 日期, 来源)`。这样 `data/latest.json` 的 `prices` 数组里，每个
  「航班+日期」组合最多会出现两条记录（携程一条、去哪儿一条），`updated_at`
  仍取全局最大 `ts`。

这是支撑"并列展示 + 高亮最低价"最小且自然的改动：不引入新的数据形状，只是把
原本"按航班+日期取一条"放宽成"按航班+日期+来源各取一条"。

## 2. 抓取实现（`scripts/fetch_prices.py`）

- 现有 `search_date` 改名为携程专用实现（如 `search_date_ctrip`），新增
  `search_date_qunar`，各自产出带正确 `source` 字段的记录。
- `fetch_all` 按以下顺序执行：完整跑一轮所有「日期 × 航线」的携程查询，再完整
  跑一轮去哪儿查询。两个来源各自独立失败、互不影响，日志按来源分段打印
  （如 `--- 携程 / 2026-07-24 ---` / `--- 去哪儿 / 2026-07-24 ---`）。
- **第一步是探测页面结构，不是直接写抓取逻辑**：去哪儿国际航班搜索结果页的
  URL 格式、DOM 结构、价格文本格式（是否也是「¥1,234起」这种形式）现在都未知，
  需要先写一个类似 `debug_page.py` 的诊断脚本截图、打印页面结构，摸清楚之后
  才能写出可靠的选择器和价格解析正则。`debug_page.py` 本身是诊断工具，不在
  正常数据流程中，可以直接复用其结构改造或新增一份。
- `source` 固定写死为 `"去哪儿"`；价格口径与携程一致：人民币·含税·含燃油·
  单程经济舱最低价，复用现有 `parse_price` 的合理范围过滤（500–50000）。

## 3. 前端展示（`index.html`）

### 3.1 「当前最低价」表格

每个「航班 × 日期」格子最多显示两行价格，**按金额从低到高动态排序**：

- 更低的一行：放大字号、绿色高亮、挂「最低」徽标
- 另一行：保持普通灰色小字，旁边标来源徽标

若该组合只抓到一个来源的价格，则照旧只显示一行 + 来源徽标，**不出现「最低」
标记**（避免单值场景下的误导性对比）。

新增 `.source-qn` 来源徽标样式（青色系，与现有 Google Flights 绿、携程棕、
飞猪紫区分开），`sourceBadge()` 函数增加对「去哪儿」/「qunar」字样的识别。

### 3.2 「历史价格趋势」折线图

数据集从「每航班一条线」拆分为「每航班 + 来源一条线」：同一航班的两条线沿用
同一主色（`COLORS` 不变），用不同线型区分来源——实线 = 携程、虚线（`borderDash`）
= 去哪儿；图例文案相应改为「SQ869 · 携程」「SQ869 · 去哪儿」这种形式。

线条数量翻倍后视觉会更挤，但 Chart.js 自带"点击图例可隐藏/显示对应数据集"
的能力可以缓解这一问题，不需要额外开发筛选 UI。
